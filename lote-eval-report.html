LotE Workout Companion — codebase evaluation

Verdict

This is an iOS-first app that is genuinely well set up for iOS, sitting on top of a game-logic core with serious bugs, with an Android target that cannot build at all.

The content work is impressively complete — the 160-workout matrix is exactly 8×5×4 with real exercises (verified programmatically), 11 of GEMINI.md’s 14 implementation guidelines are genuinely implemented, and persistence is mostly careful. But the progression loop everything hangs on is broken in several load-bearing places, the crystal economy is trivially mintable, and the app’s stated differentiator — the psychology profiles — is mostly stubs. Accessibility is entirely absent, which for this product’s audience is a feature gap, not a polish item.

CRIT 01Android cannot build — resolved minSdk 24 is below the health plugin’s required 26; manifest merge fails on any flutter build apk. CRIT 02Android would crash at launch even if it built — MainActivity extends FlutterActivity, but health 13.x casts the activity to ComponentActivity at plugin attach. CRIT 03checkBadges() is never called from anywhere — 12 of 27 displayed badges are permanently unreachable dead code.

Suggested fix order

  1. Restore game-loop integrity models · managers · economy

    Set requiredMinutes on generated quests; make nutrition-quest completion actually match meal quests; stamp lastActiveDate only on activity so streaks can grow; zero monthly-challenge progress on rollover; wire checkBadges() into the reward path.

  2. Decide Android’s fate android

    Either do the platform work — minSdk 26, FlutterFragmentActivity, the full Health Connect manifest block, INTERNET permission, real release signing — or officially declare iOS-only and delete the pretense.

  3. Make health data trustworthy managers

    Platform-guard APPLE_STAND_TIME (iOS-only; consider APPLE_STAND_HOUR instead); fix the seconds-summed-as-hours stand metric; drop the fabricated steps-derived fallbacks; persist authorization and fetch on startup/resume.

  4. Bundle Orbitron & Exo 2 as assets app shell

    One change fixes ~35 dead fontFamily strings, offline first launch, the Android-release font regression, and the privacy disclosure question.

  5. Close the economy faucets economy

    Fix the re-armable weight-goal reward; cap or cool-down the self-report micro-rewards; make monthly-challenge units consistent; give levels 51–100 something to display.

  6. Put a floor under regressions hygiene

    Pin the Flutter version in ci_post_clone.sh; add flutter analyze + flutter test to CI; stamp CI_BUILD_NUMBER; write unit tests for the manager and models first — most bugs in this report live in pure Dart that is cheap to test.

  7. Run an accessibility pass accessibility

    Contrast-safe text tokens per element theme; reduced-motion support; Semantics on the custom-painted sprite, gauges and modals; text-scale-safe layouts. For this product’s audience this is core functionality.

The android/ directory is an untouched Flutter template sitting on a modern, internally consistent Gradle stack (Gradle 9.1, AGP 9.0.1, Kotlin 2.3.20, versioning correctly wired from pubspec). CriticalconfigverifiedResolved minSdk (flutter.minSdkVersion = 24, or 21 on older Flutter) is below the minSdk 26 declared by the health 13.3.1 Android library, so the Android build fails at manifest mergecross_platform_build/android/app/build.gradle.kts:22

cross_platform_build/android/app/build.gradle.kts line 22 sets `minSdk = flutter.minSdkVersion`, which is the stock template default (24 in current Flutter stable per FlutterExtension.kt, 21 in older releases). The health plugin's android/build.gradle (verified in carp-dk/carp-health-flutter, the repo behind health 13.3.1) declares `minSdkVersion 26` with `androidx.health.connect:connect-client:1.2.0-alpha02`. AGP's manifest merger rejects an app minSdk lower than a library's ('uses-sdk:minSdkVersion 24 cannot be smaller than version 26 declared in library [cachet.plugins.health]'), so `flutter build apk` / any Android assemble task fails outright. Fix: set `minSdk = 26` explicitly in defaultConfig. No local.properties or override is present in the repo.

Verifier’s note

Only a minor scope note: the repo's CI (Xcode Cloud) builds iOS only, so the existing pipeline is unaffected; the failure hits any local/other Android build (flutter build apk, assemble tasks). The finding itself is accurate as stated.
